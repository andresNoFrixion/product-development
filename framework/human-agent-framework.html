Product Development Human-Agent Framework

Signal to Delivery — 9 Specialized Agents, 7 Artefact Types

What is this?

A complete operating system for human-agent product development — where every stage of the product lifecycle, from the first customer signal to the shipped feature, is handled by a specialised AI agent paired with a human decision-maker.

Most teams bolt AI onto existing workflows as an afterthought: a copilot here, a summariser there. This framework takes the opposite approach. It redesigns the entire product development process around the assumption that every role has an AI counterpart. The result is a structured pipeline of 9 narrow-scope agents, connected by 7 typed artefacts, where humans retain full decision authority at every stage.

The key insight is stateless handoff. Agents don't share memory or context with each other — the artefact is the only bridge between stages. Each artefact is written in a dual format: human-readable for review and approval, and machine-oriented (TOON — Token-Oriented Object Notation) so the next agent can consume it without lossy interpretation. This means any agent can be swapped, upgraded, or replaced without breaking the chain.

The framework covers two phases. Product Discovery captures opportunity signals from three customer-facing channels, converges them at a CPO decision point, then deepens them through research and solution design. Product Delivery fans out into parallel frontend development and test specification tracks, then converges at integration for release. Every transition is explicit, every decision is traced, and every artefact is versioned.

Agents are specialists

Each agent has a narrow mission, defined capabilities, and hard boundaries. No generalists. Narrow scope produces higher quality output and makes agents easier to build, test, and improve independently.

Artefacts are the contract

The typed artefact chain is the system's backbone. Each artefact has a schema, a producer, and a consumer. Walk any artefact forward or backward to trace decisions from signal to shipped code.

Humans hold the pen

Agents propose, humans decide. The CPO is the strategic convergence point. Product designers approve research and solutions. Engineers approve implementation. No autonomous shipping.

Design Principles

Specialized Agents

Narrow scope = better output. Each agent is a specialist, not a generalist.

Dual-Format Artefacts

Single TOON source that renders to human-readable view. One source of truth.

Stateless Handoff

Artefact carries ALL context. Agents are stateless between stages.

Human Decides

Agent proposes, human approves. Every decision point has a human in the loop.

Full Traceability

Walk any artefact forward or backward through the entire chain.

Product Discovery Flow

Opportunity Space → Strategy Convergence → Problem Space → Solution Space

Opportunity Space — Signal Capture

Three parallel lanes capture signals from different sources. Same output schema, different extraction specialization.
Sales / Account Manager → Agentic Account Manager Commercial context, competitive displacement, deal signals
Customer Support → Agentic Customer Support Bug vs opportunity, workaround detection, ticket frequency
Customer Success → Agentic Customer Success Relationship patterns, churn risk, expansion opportunities
Output: Opportunity Signal Human readable + AI Oriented (TOON)
↓

Strategy Convergence — CPO Funnel

All signals converge here. CPO makes proceed/park/reject decisions. Fan-in: many signals → one brief.
OPP Signal (Sales) OPP Signal (Support) OPP Signal (CS) → CPO ↔ PM Consolidation Agent
Output: Strategic Brief Consolidated, prioritized, strategy-validated
↓

Problem Space — Analysis & Research

Product Designer leads problem framing. PM Research Agent provides deep research across regulatory, technical, and market dimensions.
Strategic Brief → Product Designer ↔ PM Research Agent ↔ research/
Output: Problem Analysis Problem framing, research findings, recommendation
↓

Solution Space — Design

Product Designer makes design decisions. UX/UI Agent produces flows, component specs, interaction patterns. This is the last Discovery stage before Delivery splits.
Problem Analysis → Product Designer ↔ UX/UI Design Agent ↔ designs/
Output: Solution Design Specs, flows, acceptance criteria — SPLITS into two delivery tracks

Product Delivery Flow

Solution Design fans out into two parallel tracks, then converges at Integration.

Fan-Out: Solution Design splits into two tracks

Solution Design → Track A: Frontend Development
Solution Design → Track B: Acceptance + Test Cases

Track A: Frontend Development

Product Engineer leads implementation. Agent handles component dev, API integration.
Product Engineer ↔ SW Frontend Agent
Output: Implementation Artefact

Track B: Acceptance + Test Cases

QA Engineer leads test design. Agent generates test cases, boundary analysis, failure modes.
QA Engineer ↔ SW QA Agent
Output: Test Specification
↓ Fan-In: Both tracks converge ↓

Integration Development

Both artefacts merge here. Product Engineer resolves deviations, agent validates integration and assesses release readiness.
Implementation Artefact Test Specification → Product Engineer ↔ SW Integration Agent
Output: Integration Package → Deployment Phase — TBD

Example Thread: FX Margin Feature

One feature traced through all 7 artefact types — from initial customer signal to release-ready package.

OPP-2026-0042 Opportunity Signal captured
Source Sales call — Account Manager
Customer FinTech Corp (Mid-market, €2M+ monthly cross-border)
Signal "Three enterprise clients independently asked about applying custom FX spreads on cross-border payments this quarter"
Similar Signals 6 related in 90 days across Sales, Support, CS
Strategic Alignment HIGH — Revenue Diversification pillar
Estimated Impact €180K ARR potential, medium effort
Confidence 0.82
↓ many:1 — 7 signals cluster into 1 brief
SB-2026-0015 Strategic Brief approved
Signal Cluster 7 signals, 3 sources (Sales 3, Support 2, CS 2), 90-day span
Pattern Enterprise clients need configurable FX margins on international payments
CPO Decision PROCEED — Priority P1
Dependencies Banking Circle FX API (held rates, RFQ)
Problem Hypothesis "Businesses cannot monetize international payment flows because there is no mechanism to apply custom FX markup"
Scope Guidance Fixed margin per currency pair; defer tiered/dynamic pricing to Phase 2
↓
PA-2026-0015 Problem Analysis validated
Problem Statement Businesses cannot apply custom markup on FX rates for international payments, losing a key revenue stream
Regulatory Findings PSD2 requires pre-transaction FX cost disclosure; CBPR2 mandates total charges transparency; B2B opt-out available
Technical Findings Banking Circle supports held rates (30s TTL) + RFQ; margin must be applied before rate lock
Market Findings Currencycloud offers 0-3% configurable spread; Wise Business uses transparent flat fee model
Recommendation Proceed — Fixed basis-point margin per currency pair, applied at rate quote time
Open Questions Rounding policy for sub-pip amounts? Display format for end-customer transparency?
↓
SD-2026-0015 Solution Design approved
Solution FX Margin Configuration panel in Banking Portal + API margin parameter
Approach Apply margin as basis-point spread at held-rate request time
Scope In Fixed margin per pair, margin preview, audit trail
Scope Out Tiered pricing, dynamic margins, volume discounts (Phase 2)
Key User Flow Admin → Portal → FX Settings → Set margin per currency pair → Save → Payment quotes include margin → Customer sees breakdown
Acceptance Criteria AC-1: Margin accurate to 4dp  |  AC-2: Pre-transaction breakdown displayed  |  AC-3: Changes audit-logged  |  AC-4: API supports margin_override
↙ Track A ↘ Track B
IA-2026-0015 Implementation ready
Components MarginConfigPanel, RatePreviewWidget, MarginAuditLog — all DONE
API Integrations BC Held Rate wrapped with margin layer; new POST /settings/fx-margins
Architecture Decision Margin applied server-side before rate returned (security); config cached 60s TTL
Deviations None
TS-2026-0015 Test Specification complete
Coverage 94% of acceptance criteria mapped to test cases
Sample Tests TC-001: 50bps on EUR/USD 1.0842 → expect 1.0892
TC-005: Display "Base: 1.0842 | Margin: +0.50% | Final: 1.0896"
Risk Area Rate expiry during margin application (30s TTL edge case)
↓ fan-in — both tracks converge
IP-2026-0015 Integration Package ready
Merge Status Implementation + Test Spec aligned — 0 deviations
Integration Tests 47 passed, 0 failed, 0 blocked
Release Readiness READY
Rollback Plan Feature flag fx_margin_enabled → set to false
Release Notes Configurable FX margin per currency pair via Portal and API. Fixed margin only (tiered pricing planned Phase 2). Requires Banking Circle held-rate API v2.1+
